# 02_PRODUCT_PRINCIPLES.md — 무엇을 만들지 판단하는 기준

새 기능 아이디어가 나왔을 때, 코덱스/Claude/크리스님 누구든 이 체크리스트로
먼저 거른다. 이건 매번 새로 토론하지 않기 위한 고정된 필터다.

## 통과해야 하는 5가지 질문

1. **이게 리더가 "오늘 누구를 챙길지" 판단하는 데 직접 도움이 되는가?**
   안 되면 — 아무리 멋있어도 보류한다.

2. **기존 데이터로 만들 수 있는가, 새 데이터가 필요한가?**
   기존 데이터(checkins, prayers, care_messages)로 다른 방식으로
   보여주는 것이면 우선순위가 높다. 새 테이블/새 입력 양식이 필요하면
   한 단계 더 신중하게 검토한다 (01_NORTH_STAR.md의 두 축에 정확히
   해당하는지 다시 확인).

3. **이걸 채팅이나 AI 자동화로 풀려고 하고 있지 않은가?**
   "차별성/사업성 때문에 카톡 써라"처럼 회피하는 것도 금지지만,
   반대로 "AI가 대신 판단하게 하자"도 금지다. 둘 다 01_NORTH_STAR.md의
   금지 목록에 있다. 사람(리더)이 보고 판단하게 돕는 도구인지 항상 확인.

4. **멤버와 리더 둘 다에게 공정한가?**
   리더 전용 기능(컨택 로그, 키워드 신호 등)은 멤버 화면에 절대
   노출되면 안 된다. 반대로 멤버가 나누는 내용이 리더에게 과도하게
   감시처럼 느껴지는 형태가 아닌지 점검한다.

5. **오늘 하루 안에 검증까지 끝낼 수 있는 크기인가?**
   안 되면 더 작은 단위로 쪼갠다. 한 티켓에 여러 화면, 여러 테이블을
   동시에 바꾸지 않는다.

## 디자인 판단 기준

- 새 색상 체계를 만들지 않는다. 기존 leaf/clay/blue 톤만 재사용.
- 경고/위험 표시는 항상 절제된 디자인으로. 빨간색 남발, 점수/퍼센트
  노출 금지 (02 원칙 3번과 연결).
- "더 모던하게"같은 추상적 디자인 지시는 받지 않는다. 구체적인 질문
  (예: "3초 안에 누구를 챙길지 알 수 있는가?")으로 바꿔서 판단한다.

## 검증 기준

- "완료"는 명령어 raw 출력 또는 실제 클릭 검증으로만 인정한다.
- 보고와 실제 git/DB 상태가 다르면, 보고를 신뢰하지 않고 직접 검증한다.
- 한 티켓이 다음 티켓의 전제 조건이면, 앞 티켓이 검증되기 전에
  뒤 티켓을 시작하지 않는다.
