# Workflow 02: Feature Discovery

## Trigger

A new idea, user request, beta observation, or vague improvement request appears.

## Goal

Understand the problem and decide whether the idea deserves a feature meeting, manual test, or deferral.

## Required Input

- Proposed feature or observed problem.
- Who experienced it: leader, member, or operator.
- Evidence source: user quote, beta observation, internal assumption, or bug.
- Urgency.

## Agents To Call

- Entrepreneur
- Designer
- Marketer

## Skills To Use

- brainstorm
- competitor-differentiation
- product-plan

## Step-By-Step Process

1. Restate the problem without assuming the solution.
2. Brainstorm possible responses, including non-feature options.
3. Run the competitor differentiation skill.
4. Filter out ideas that imitate the comparison products.
5. Choose build now, test manually, defer, reject, or escalate to feature meeting.
6. Draft a feature brief only if the idea passes the decision gate.

## Decision Gate

Escalate to feature meeting only if the idea directly strengthens daily check-in, prayer request cards, "기도했어요", leader care, or public beta readiness.

## Files To Update

- `superpowers/templates/feature-brief.md` copy for approved exploration.
- `superpowers/templates/decision-record.md` copy for rejected or deferred scope decisions.
- `docs/PROJECT_STATUS.md` if the roadmap changes.

## Final Output Format

```md
## Feature Discovery

Problem:
- ...

Best Options:
- ...

Differentiation Decision:
- ...

Next Step:
- Feature meeting / Manual test / Defer / Reject
```
