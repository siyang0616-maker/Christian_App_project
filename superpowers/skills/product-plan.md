# Product Plan Skill

## Purpose

Turn an approved direction into a short product plan that protects the MVP scope.

## When To Use

- Before writing a feature brief.
- When deciding what belongs in the next beta iteration.
- When updating docs such as `docs/PROJECT_STATUS.md` or `docs/MVP_SCOPE.md`.

## Inputs Required

- Product goal.
- Target user.
- Problem statement.
- Success signal.
- Scope constraints.

## Step-By-Step Process

1. State the user and problem.
2. Define the desired behavior change.
3. Explain why this supports daily check-in, prayer memory, or leader care.
4. List in-scope and out-of-scope items.
5. Define a beta success signal.
6. Identify docs that should be updated after the decision.

## Output Format

```md
## Product Plan

Goal:
- ...

Target User:
- ...

In Scope:
- ...

Out Of Scope:
- ...

Success Signal:
- ...

Docs To Update:
- ...
```

## Acceptance Criteria

- Keeps the MVP low-cost and mobile-first.
- Names out-of-scope items clearly.
- Includes a measurable or observable beta signal.
- Does not add app implementation details beyond product intent.
